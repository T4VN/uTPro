# 4.3 uTPro Custom Configuration

All uTPro-specific settings are under the `uTPro` section in `appsettings.json`:

```json
{
  "uTPro": {
    "Backoffice": {
      "Enabled": true,
      "Url": "bo.yourdomain.com"
    },
    "RememberLanguage": {
      "Enabled": true,
      "ListExludeRequestLanguage": {
        "Enabled": true,
        "Paths": []
      }
    }
  }
}
```

| Key | Description |
|-----|-------------|
| `uTPro:Backoffice:Enabled` | Enable separate backoffice domain |
| `uTPro:Backoffice:Url` | Backoffice domain(s), comma-separated |
| `uTPro:RememberLanguage:Enabled` | Enable cookie-based language persistence |
| `uTPro:RememberLanguage:ListExludeRequestLanguage:Enabled` | Enable path exclusion for language detection |
| `uTPro:RememberLanguage:ListExludeRequestLanguage:Paths` | Array of paths to exclude from language detection |

# 4.4 Security Configuration

uTPro includes built-in security hardening configured in `Startup/SecuritySetup.cs` and `Startup/PipelineSetup.cs`:

- **Security Headers** (production): `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`, `Strict-Transport-Security`
- **Request Size Limits**: 128MB max upload, 4MB per form value, 10240 max form values
- **Sync IO disabled**: Both IIS and Kestrel have `AllowSynchronousIO = false` to prevent thread-pool starvation under load
- **Data Protection**: Keys persisted with 90-day rotation, configurable shared path for load-balanced deployments
- **Domain Policy**: Configurable domain allowlist with wildcard support in `CheckPolicy.cs`
- **HSTS**: Enabled in production with 1-year max-age and includeSubDomains
- **Kestrel**: Server header disabled

# 4.5 Performance Configuration

Performance is configured in `Startup/PerformanceSetup.cs` and `Startup/PipelineSetup.cs`:

### Minification & Compression
- **WebOptimizer**: CSS/JS minification for `css/**/*.css`, `assets/**/*.css`, `js/**/*.js`, `assets/**/*.js`, `scripts/**/*.js`
- **WebOptimizer forced in Development**: `webOptimizer.enableCaching: true` in `appsettings.Development.json` ensures minification runs in all environments
- **WebMarkupMin**: HTML minification + Brotli/GZip/Deflate compression (Brotli prioritized, 10MB max response size)
- **Statistics disabled** in production to save CPU
- **Pipeline order**: `UseWebOptimizer()` → `UseStaticFiles()` → `UseWebMarkupMin()` (critical for minification to work)

### Caching (multi-layer)
- **Output Cache** (`AddOutputCache`): Stores fully-rendered HTML responses in server memory. Subsequent requests skip Razor + Umbraco entirely (0ms server processing). Policies:
  - `Page` — 120s TTL, varies by host/query/Accept-Language, tagged `"pages"` for eviction
  - `Sitemap` — 30min TTL, varies by host, tagged `"sitemap"`
- **Browser Cache** (`Cache-Control` header): Website pages get `public, max-age=120, stale-while-revalidate=60` — browser shows cached page instantly while revalidating in background
- **Static File Cache**:
  - Assets in `/assets/`, `/css/`, `/scripts/`, `/js/` → `public, max-age=31536000, immutable` (1 year)
  - Media uploads in `/uploads/`, `/media/` → `public, max-age=604800, stale-while-revalidate=86400` (7 days)
- **Sitemap Memory Cache** (`IMemoryCache`): XML sitemap cached 30 minutes in `SitemapFoundation`
- **Response Caching**: Adds `Vary` headers for downstream CDN/proxy validation

### Razor
- **Runtime Compilation**: Only enabled in Development environment (saves ~200MB RAM in production)
- **Precompiled views**: `RazorCompileOnBuild=true` in csproj ensures views are compiled at build time

### Request Buffering
- **Global buffering disabled**: `BufferBody = false` — no longer copies every request body into memory
- **Selective buffering**: Only endpoints that need to re-read the body should enable it explicitly

# 4.7 Multi-site IIS hosting (load balancing) — moving `umbraco/Data/TEMP` out of the project

When you host one backoffice + multiple public sites on IIS that share the same SQL database, do **not** try to share `umbraco/Data/TEMP` between them. NuCache, Examine indexes and `MainDom` all take exclusive file locks, so a shared folder causes boot failures and corrupt indexes.

The correct pattern is the same one Umbraco documents for [Azure Web Apps load balancing](https://docs.umbraco.com/umbraco-cms/fundamentals/setup/server-setup/load-balancing/azure-web-apps):

- Each site keeps its own TEMP folder, placed in `%TEMP%` (outside the project).
- Content changes propagate via `DatabaseServerMessenger` (polls the shared SQL).
- The backoffice site is pinned as `SchedulingPublisher`; public sites are `Subscriber`s.

## 4.7.1 Baseline settings (apply to every site)

Add these three settings to every site's `appsettings.json`. The template already ships them configured:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "MainDomLock": "FileSystemMainDomLock"
      },
      "Hosting": {
        "LocalTempStorageLocation": "EnvironmentTemp"
      },
      "Examine": {
        "LuceneDirectoryFactory": "TempFileSystemDirectoryFactory"
      }
    }
  }
}
```

| Key | Effect |
|---|---|
| `Hosting:LocalTempStorageLocation: EnvironmentTemp` | Umbraco writes NuCache, locks and other TEMP files to `%TEMP%/UmbracoData/<hash-of-ContentRoot>/`. Each site gets its own folder automatically, nothing lands in the project. |
| `Examine:LuceneDirectoryFactory: TempFileSystemDirectoryFactory` | Lucene indexes are created under the same `%TEMP%` location. Use `SyncedTempFileSystemDirectoryFactory` instead on the single-instance backoffice if you want indexes persisted and synced on restart. |
| `Global:MainDomLock: FileSystemMainDomLock` | File-based MainDom lock. Works across multiple processes / instances on Windows where the default semaphore is machine-wide and breaks multi-site hosting. |

On IIS make sure the app pool identity has write access to `%TEMP%`. With `ApplicationPoolIdentity` that is `C:\Windows\Temp` by default (enable "Load User Profile" only if you actually need a per-profile temp).

> Media uploads (`uploads/`) are controlled separately via `Umbraco:CMS:Global:UmbracoMediaPath` and are not affected by these settings.

### 4.7.1.1 Override `%TEMP%` from `appsettings.json` (code-driven alternative)

If you don't want to touch `web.config` on every deploy, this template provides a source-code override in `Startup/HostingSetup.cs`. Set `uTPro:Hosting:TempPath` and the app will set the process-level `TMP` / `TEMP` environment variables at startup, before Umbraco reads `Path.GetTempPath()`:

```json
{
  "uTPro": {
    "Hosting": {
      "TempPath": "D:\\UmbracoTemp\\SiteA"
    }
  }
}
```

- Empty string (the default) leaves `TMP` / `TEMP` untouched — Umbraco falls back to the OS-provided temp folder.
- Absolute paths are used as-is.
- Relative paths are resolved against `ContentRoot` (NOT the process working directory).
- The folder is auto-created on boot if missing; the app pool identity must have `Modify` on its parent.

# 4.8 Database Provider (PostgreSQL / SQL Server / SQLite)

uTPro runs on any database engine Umbraco 17 supports. The engine is selected by two keys under `ConnectionStrings` in `appsettings.json`: `umbracoDbDSN` (the connection string) and `umbracoDbDSN_ProviderName` (the ADO.NET provider).

uTPro ships configured for **PostgreSQL** out of the box — `uTPro.Project.Web` references the `Our.Umbraco.PostgreSql` package and both `appsettings.json` / `appsettings.Development.json` default to the `Npgsql2` provider. Switching to SQL Server or SQLite requires only changing the two connection-string keys below; no code changes.

| Engine | `umbracoDbDSN_ProviderName` | Extra package required |
|--------|-----------------------------|------------------------|
| PostgreSQL (default) | `Npgsql2` | `Our.Umbraco.PostgreSql` (pre-referenced) |
| SQL Server | `Microsoft.Data.SqlClient` | none (built into Umbraco) |
| SQLite | `Microsoft.Data.Sqlite` | none (built into Umbraco) |

### PostgreSQL (default)

Create an empty PostgreSQL database, then point the connection string at it:

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=localhost;Port=5432;Database=uTPro;User Id=postgres;Password=postgres;",
    "umbracoDbDSN_ProviderName": "Npgsql2"
  }
}
```

### SQL Server

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=(local);Database=uTPro;Integrated Security=true;TrustServerCertificate=true;",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.SqlClient"
  }
}
```

### SQLite

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Data Source=|DataDirectory|/Umbraco.sqlite.db;Cache=Shared;Foreign Keys=True;Pooling=True",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.Sqlite"
  }
}
```

> **Note:** The `Our.Umbraco.PostgreSql` package is a community provider ([Our.Umbraco.PostgreSql on NuGet](https://www.nuget.org/packages/Our.Umbraco.PostgreSql)). Its provider name is `Npgsql2` (not `Npgsql`) — this is intentional, so it doesn't clash with the stock `Npgsql` ADO.NET factory. Keep its version aligned with your Umbraco major version (e.g. `17.x` for Umbraco 17). Never mix providers in the same environment — pick one engine per deployment. The default credentials (`postgres`/`postgres`) are for local development only; use a dedicated least-privilege role and a secret store for production. Leaving an empty `umbracoDbDSN` triggers the Umbraco install wizard, which lets you choose the engine from the UI instead.
