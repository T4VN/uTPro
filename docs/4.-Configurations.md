# 4.7 Multi-site IIS hosting (load balancing) — moving `umbraco/Data/TEMP` out of the project

When you host one backoffice + multiple public sites on IIS that share the same SQL database, do **not** try to share `umbraco/Data/TEMP` between them. NuCache, Examine indexes and `MainDom` all take exclusive file locks, so a shared folder causes boot failures and corrupt indexes.

The correct pattern is the same one Umbraco documents for [Azure Web Apps load balancing](https://docs.umbraco.com/umbraco-cms/fundamentals/setup/server-setup/load-balancing/azure-web-apps):

- Each site keeps its own TEMP folder, placed in `%TEMP%` (outside the project).
- Content changes propagate via `DatabaseServerMessenger` (polls the shared SQL).
- The backoffice site is pinned as `SchedulingPublisher`; public sites are `Subscriber`s.

## 4.7.1 Baseline settings (apply to every site)

Add these three settings to every site's `appsettings.json`. The template already ships them configured:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "MainDomLock": "FileSystemMainDomLock"
      },
      "Hosting": {
        "LocalTempStorageLocation": "EnvironmentTemp"
      },
      "Examine": {
        "LuceneDirectoryFactory": "TempFileSystemDirectoryFactory"
      }
    }
  }
}
```

| Key | Effect |
|---|---|
| `Hosting:LocalTempStorageLocation: EnvironmentTemp` | Umbraco writes NuCache, locks and other TEMP files to `%TEMP%/UmbracoData/<hash-of-ContentRoot>/`. Each site gets its own folder automatically, nothing lands in the project. |
| `Examine:LuceneDirectoryFactory: TempFileSystemDirectoryFactory` | Lucene indexes are created under the same `%TEMP%` location. Use `SyncedTempFileSystemDirectoryFactory` instead on the single-instance backoffice if you want indexes persisted and synced on restart. |
| `Global:MainDomLock: FileSystemMainDomLock` | File-based MainDom lock. Works across multiple processes / instances on Windows where the default semaphore is machine-wide and breaks multi-site hosting. |

On IIS make sure the app pool identity has write access to `%TEMP%`. With `ApplicationPoolIdentity` that is `C:\Windows\Temp` by default (enable "Load User Profile" only if you actually need a per-profile temp).

> Media uploads (`uploads/`) are controlled separately via `Umbraco:CMS:Global:UmbracoMediaPath` and are not affected by these settings.

### 4.7.1.1 Override `%TEMP%` from `appsettings.json` (code-driven alternative)

If you don't want to touch `web.config` on every deploy, this template provides a source-code override. Set `uTPro:Hosting:TempPath` and `Program.cs` will set the process-level `TMP` / `TEMP` environment variables at startup, before Umbraco reads `Path.GetTempPath()`:

```json
{
  "uTPro": {
    "Hosting": {
      "TempPath": "D:\\UmbracoTemp\\SiteA"
    }
  }
}
```

- Empty string (the default) leaves `TMP` / `TEMP` untouched — Umbraco falls back to the OS-provided temp folder.
- Absolute paths are used as-is.
- Relative paths are resolved against `ContentRoot` (NOT the process working directory).
- The folder is auto-created on boot if missing; the app pool identity must have `Modify` on its parent.

Combined with `Hosting:LocalTempStorageLocation = EnvironmentTemp`, each site can point to a dedicated disk/folder without editing `web.config`. Choose whichever approach fits your deployment pipeline — `web.config` `<environmentVariables>` (see 4.7.7) or this source-driven setting — but don't use both.

## 4.7.2 Explicit server roles

With a shared database, exactly one instance must handle scheduled publishing and backoffice writes. Pin roles explicitly instead of letting Umbraco elect them at runtime.

> **Important:** `ServerRole` is **not** set in the template's `appsettings.json` on purpose. Setting it in source would break dev (single-site) and single-deploy scenarios. Apply it per site at deploy time using an environment-specific file (e.g. `appsettings.Production.json`) or `web.config` environment variables. Files like `appsettings.Production.json` override `appsettings.json` when `ASPNETCORE_ENVIRONMENT=Production`.

**Backoffice site** — create `appsettings.Production.json` next to `appsettings.json` with:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "DatabaseServerRegistrar": {
          "ServerRole": "SchedulingPublisher"
        }
      }
    }
  }
}
```

**Public sites** (web1, web2, web3) — same file, value `Subscriber`:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "DatabaseServerRegistrar": {
          "ServerRole": "Subscriber"
        }
      }
    }
  }
}
```

Rule of thumb: only the backoffice runs scheduled tasks. Subscribers render pages and pull cache refresher messages from the DB.

**Alternative: set via environment variable in `web.config`** (useful if you want a single published artefact and only differ between sites via `web.config`):

```xml
<environmentVariables>
  <environmentVariable name="UMBRACO__CMS__Global__DatabaseServerRegistrar__ServerRole"
                       value="Subscriber" />
</environmentVariables>
```

ASP.NET Core's environment-variable configuration provider maps `__` (double underscore) to `:` in the config hierarchy.

## 4.7.3 Data Protection keys (shared auth across sites)

All four sites serve the same domain family, so cookies / antiforgery tokens issued by one site must be decryptable by the others. That means every site must load the same Data Protection key ring with the same `ApplicationName`.

```json
{
  "uTPro": {
    "DataProtection": {
      "KeysPath": "C:\\UmbracoKeys\\uTPro-Shared",
      "ApplicationName": "uTPro"
    }
  }
}
```

| Key | Description |
|---|---|
| `uTPro:DataProtection:KeysPath` | Absolute folder, or relative to the site's ContentRoot. Put it OUTSIDE the project so publishing doesn't touch it. |
| `uTPro:DataProtection:ApplicationName` | Must match on every site that needs to share cookies. Defaults to `uTPro`. |

Grant the IIS app pool identity `Modify` on `C:\UmbracoKeys\uTPro-Shared`. Back the folder up — losing it invalidates every existing auth cookie.

If `KeysPath` is not set, the template falls back to:
1. `%TEMP%/UmbracoData/<hash>/PersistKeys` when `LocalTempStorageLocation = EnvironmentTemp` (default in this template), or
2. `<ContentRoot>/umbraco/Data/TEMP/PersistKeys` otherwise.

## 4.7.4 Speeding up publish-to-public-site propagation

`DatabaseServerMessenger` polls the DB every 5 seconds by default; that is why web1 does not reflect a newly published node immediately after clicking Save and Publish in the backoffice. Tune the polling interval if you need faster sync:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "DatabaseServerMessenger": {
          "TimeBetweenSyncOperations": "00:00:02",
          "TimeBetweenPruneOperations": "00:01:00"
        }
      }
    }
  }
}
```

Do not drop `TimeBetweenSyncOperations` below ~1 second; every subscriber queries the DB on each tick.

## 4.7.5 Recommended disk layout

```
C:\inetpub\sites\uTPro.Backoffice\   <- published backoffice
C:\inetpub\sites\uTPro.Web1\         <- public site 1
C:\inetpub\sites\uTPro.Web2\
C:\inetpub\sites\uTPro.Web3\
C:\UmbracoKeys\uTPro-Shared\         <- shared DP keys (writable by app pools)
%TEMP%\UmbracoData\<hash-per-site>\  <- auto-created by Umbraco, per-site isolated
```

What each site needs in `appsettings.json`:

- Same `ConnectionStrings:umbracoDbDSN`.
- Same `uTPro:DataProtection:KeysPath` + `ApplicationName`.
- Baseline TEMP / Examine / MainDomLock settings from 4.7.1 (already in the template).

What each site needs **on top** of the template (deploy-time overrides, not committed to source):

- `appsettings.Production.json` with the correct `ServerRole` (`SchedulingPublisher` for backoffice, `Subscriber` for web1/web2/web3).
- Optional: `web.config` `<environmentVariables>` to redirect `TMP` / `TEMP` per site (see 4.7.7).

## 4.7.6 Deployment notes

- Publish to each site folder separately. Files / views / DLLs are **not** shared — the DB is.
- After deploying, public sites pick up content/media changes automatically via the messenger; code changes require a restart of that site's app pool.
- Do not edit Razor views on a live server to "sync" them — they live on each site's disk independently.
- The media folder (`uploads/`) needs to be shared between all four sites (UNC path, Azure Blob, or similar) so the backoffice and public sites see the same files. This is outside the scope of this template.

## 4.7.7 Redirecting `%TEMP%` per site via `web.config`

`LocalTempStorageLocation = EnvironmentTemp` tells Umbraco to use `Path.GetTempPath()`, which is `%TEMP%` on Windows. By default that resolves to `C:\Windows\Temp` for IIS app pools — shared across every site on the server. If you want each site to write its TEMP files to a different disk or folder (for example, keep it on a dedicated data drive, or isolate per site explicitly), override `TMP` and `TEMP` at the site level in `web.config`:

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <location path="." inheritInChildApplications="false">
    <system.webServer>
      <handlers>
        <add name="aspNetCore" path="*" verb="*" modules="AspNetCoreModuleV2" resourceType="Unspecified" />
      </handlers>
      <aspNetCore processPath="dotnet"
                  arguments=".\uTPro.Project.Web.dll"
                  stdoutLogEnabled="false"
                  stdoutLogFile=".\logs\stdout"
                  hostingModel="inprocess">
        <environmentVariables>
          <environmentVariable name="TMP"  value="D:\UmbracoTemp\SiteA" />
          <environmentVariable name="TEMP" value="D:\UmbracoTemp\SiteA" />
          <environmentVariable name="ASPNETCORE_ENVIRONMENT" value="Production" />
        </environmentVariables>
      </aspNetCore>
    </system.webServer>
  </location>
</configuration>
```

Repeat per site with a different folder (`D:\UmbracoTemp\SiteB`, `SiteC`, `Backoffice`). Combined with `LocalTempStorageLocation = EnvironmentTemp`, Umbraco will write:

```
D:\UmbracoTemp\SiteA\UmbracoData\<hash-of-ContentRoot>\
  ├─ NuCache\
  ├─ ExamineIndexes\
  └─ ... (MainDom lock, PersistKeys if not overridden)
```

**Prerequisites:**

- Create the folders first (`D:\UmbracoTemp\SiteA`, etc.). The app pool identity needs `Modify` on each one. For `ApplicationPoolIdentity` grant it to `IIS AppPool\<PoolName>`.
- Restart the app pool after changing `web.config` so the new environment variables take effect.
- Do not point multiple sites to the same `TMP` folder. It defeats the isolation and causes the same lock contention as sharing `umbraco/Data/TEMP`.

**Why this beats editing the system-wide `TMP`/`TEMP`:** system-level changes affect every service on the host (Windows services, scheduled tasks, other IIS sites you don't own). The `<environmentVariables>` block in `web.config` applies only to that site's dotnet process.

> `web.config` is generated automatically on `dotnet publish`. If you keep a hand-crafted `web.config` in the project root, add `<PublishWebConfigTransform>` or mark it as content so it's preserved; otherwise edit `web.config` at the deployed location after publish (and re-apply after each deploy — or script it).
