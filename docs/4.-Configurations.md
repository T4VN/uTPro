# 4.4 Security Configuration

uTPro includes built-in security hardening. Most of it is code-level (in `Startup/*.cs`), while the HTTP **security headers are content-driven** so editors can tune them per site from the backoffice without a redeploy:

- **Security Headers** ÔÇö resolved per request by `SecurityHeadersFoundation` (in `uTPro.Foundation.Middleware`) from a per-site **`GlobalSecurityHeadersSettings`** backoffice node (see [4.4.1](#441-security-headers-backoffice-driven)). When that node is absent or its `Enabled` toggle is off, no security headers are emitted.
- **Request Size Limits**: 128MB max upload, 4MB per form value, 10240 max form values
- **Sync IO disabled**: Both IIS and Kestrel have `AllowSynchronousIO = false` to prevent thread-pool starvation under load
- **Data Protection**: Keys persisted with 90-day rotation, configurable shared path for load-balanced deployments
- **Domain Policy**: Configurable domain allowlist with wildcard support in `CheckPolicy.cs`
- **Kestrel**: Server header disabled

## 4.4.1 Security headers (backoffice-driven)

Security headers are no longer hard-coded ÔÇö they are read from a `GlobalSecurityHeadersSettings` node that can be composed onto the site's **Folder Settings** node, the **site root**, or the **current page** (first match wins). Sensible code defaults fill in any field left blank on an enabled node, so turning the feature on with everything blank still gives you `nosniff`, `SAMEORIGIN`, and a 1-year HSTS on HTTPS.

| Field | Emitted header | Default when blank |
|-------|----------------|--------------------|
| `Enabled` | *(master toggle ÔÇö off means no headers at all)* | ÔÇö |
| `XContentTypeOptions` | `X-Content-Type-Options` | `nosniff` |
| `ReferrerPolicy` | `Referrer-Policy` | *(omitted)* |
| `PermissionsPolicy` | `Permissions-Policy` | *(omitted)* |
| `XFrameOptions` | `X-Frame-Options` | `SAMEORIGIN` (only when no CSP `frame-ancestors` is in play) |
| `ContentSecurityPolicyCsp` | `Content-Security-Policy` | *(omitted)* |
| `ContentSecurityPolicyReportOnly` | sends the CSP as `Content-Security-Policy-Report-Only` instead (framing stays enforced) | `false` |
| `HstsEnabled` | `Strict-Transport-Security` (HTTPS requests only) | `false` |
| `HstsMaxAgeSeconds` | `max-age=ÔÇŽ` | `31536000` (1 year) |
| `HstsIncludeSubDomains` | `; includeSubDomains` | `false` |
| `HstsPreload` | `; preload` | `false` |

**Behaviour worth knowing:**

- **Clickjacking protection is always enforced** on an enabled node. When a backoffice domain is configured (`uTPro:Backoffice:Domain`), the trusted backoffice host is added to a CSP `frame-ancestors 'self' ÔÇŽ` directive so cross-domain backoffice **preview keeps working**; otherwise `X-Frame-Options` is used. An explicit CSP that omits `frame-ancestors` gets the backoffice host appended automatically.
- **Header-injection safe**: every value is collapsed to a single line (CR/LF/tabs/multiple spaces Ôćĺ one space) before being emitted, so editors can format a CSP one directive per line without risking response splitting (CWE-113).
- **Fails open**: if the settings/content lookup errors, the request is served with no security headers (and a warning is logged) rather than returning a 500 ÔÇö a content issue can never take the frontend down.

# 4.5 Performance Configuration

Performance is configured in `Startup/PerformanceSetup.cs` and `Startup/PipelineSetup.cs`:

### Minification & Compression
- **WebOptimizer**: CSS/JS minification for `css/**/*.css`, `assets/**/*.css`, `js/**/*.js`, `assets/**/*.js`, `scripts/**/*.js`
- **WebOptimizer forced in Development**: `webOptimizer.enableCaching: true` in `appsettings.Development.json` ensures minification runs in all environments
- **WebMarkupMin**: HTML minification + Brotli/GZip/Deflate compression (Brotli prioritized, 10MB max response size)
- **Statistics disabled** in production to save CPU
- **Pipeline order**: `UseWebOptimizer()` Ôćĺ `UseStaticFiles()` Ôćĺ `UseWebMarkupMin()` (critical for minification to work)

### Caching (multi-layer)
- **Output Cache** (`AddOutputCache`): Stores fully-rendered HTML responses in server memory. Subsequent requests skip Razor + Umbraco entirely (0ms server processing). Policies:
  - `Page` ÔÇö 120s TTL, varies by host/query/Accept-Language, tagged `"pages"` for eviction
  - `Sitemap` ÔÇö 30min TTL, varies by host, tagged `"sitemap"`
- **Browser Cache** (`Cache-Control` header): Website pages get `public, max-age=120, stale-while-revalidate=60` ÔÇö browser shows cached page instantly while revalidating in background
- **Static File Cache**:
  - Assets in `/assets/`, `/css/`, `/scripts/`, `/js/` Ôćĺ `public, max-age=31536000, immutable` (1 year)
  - Media uploads in `/uploads/`, `/media/` Ôćĺ `public, max-age=604800, stale-while-revalidate=86400` (7 days)
- **Sitemap Memory Cache** (`IMemoryCache`): XML sitemap cached 30 minutes in `SitemapFoundation`
- **Response Caching**: Adds `Vary` headers for downstream CDN/proxy validation

### Razor
- **Runtime Compilation**: Only enabled in Development environment (saves ~200MB RAM in production)
- **Precompiled views**: `RazorCompileOnBuild=true` in csproj ensures views are compiled at build time

### Request Buffering
- **Global buffering disabled**: `BufferBody = false` ÔÇö no longer copies every request body into memory
- **Selective buffering**: Only endpoints that need to re-read the body should enable it explicitly

# 4.7 Multi-site IIS hosting (load balancing) ÔÇö moving `umbraco/Data/TEMP` out of the project

When you host one backoffice + multiple public sites on IIS that share the same SQL database, do **not** try to share `umbraco/Data/TEMP` between them. NuCache, Examine indexes and `MainDom` all take exclusive file locks, so a shared folder causes boot failures and corrupt indexes.

The correct pattern is the same one Umbraco documents for [Azure Web Apps load balancing](https://docs.umbraco.com/umbraco-cms/fundamentals/setup/server-setup/load-balancing/azure-web-apps):

- Each site keeps its own TEMP folder, placed in `%TEMP%` (outside the project).
- Content changes propagate via `DatabaseServerMessenger` (polls the shared SQL).
- The backoffice site is pinned as `SchedulingPublisher`; public sites are `Subscriber`s.

## 4.7.1 Baseline settings (apply to every site)

Add these three settings to every site's `appsettings.json`. The template already ships them configured:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "MainDomLock": "FileSystemMainDomLock"
      },
      "Hosting": {
        "LocalTempStorageLocation": "EnvironmentTemp"
      },
      "Examine": {
        "LuceneDirectoryFactory": "TempFileSystemDirectoryFactory"
      }
    }
  }
}
```

| Key | Effect |
|---|---|
| `Hosting:LocalTempStorageLocation: EnvironmentTemp` | Umbraco writes NuCache, locks and other TEMP files to `%TEMP%/UmbracoData/<hash-of-ContentRoot>/`. Each site gets its own folder automatically, nothing lands in the project. |
| `Examine:LuceneDirectoryFactory: TempFileSystemDirectoryFactory` | Lucene indexes are created under the same `%TEMP%` location. Use `SyncedTempFileSystemDirectoryFactory` instead on the single-instance backoffice if you want indexes persisted and synced on restart. |
| `Global:MainDomLock: FileSystemMainDomLock` | File-based MainDom lock. Works across multiple processes / instances on Windows where the default semaphore is machine-wide and breaks multi-site hosting. |

On IIS make sure the app pool identity has write access to `%TEMP%`. With `ApplicationPoolIdentity` that is `C:\Windows\Temp` by default (enable "Load User Profile" only if you actually need a per-profile temp).

> Media uploads (`uploads/`) are controlled separately via `Umbraco:CMS:Global:UmbracoMediaPath` and are not affected by these settings.

### 4.7.1.1 Override `%TEMP%` from `appsettings.json` (code-driven alternative)

If you don't want to touch `web.config` on every deploy, this template provides a source-code override in `Startup/HostingSetup.cs`. Set `uTPro:Hosting:TempPath` and the app will set the process-level `TMP` / `TEMP` environment variables at startup, before Umbraco reads `Path.GetTempPath()`:

```json
{
  "uTPro": {
    "Hosting": {
      "TempPath": "D:\\UmbracoTemp\\SiteA"
    }
  }
}
```

- Empty string (the default) leaves `TMP` / `TEMP` untouched ÔÇö Umbraco falls back to the OS-provided temp folder.
- Absolute paths are used as-is.
- Relative paths are resolved against `ContentRoot` (NOT the process working directory).
- The folder is auto-created on boot if missing; the app pool identity must have `Modify` on its parent.

## 4.7.2 Server roles (`uTPro:Hosting:ServerRole`)

In a load-balanced setup every app runs against the **same database**, so Umbraco needs to know which one owns the scheduled work (publishing scheduled content, keep-alive, health checks, log/temp cleanup). That is the **server role** ÔÇö enum `Umbraco.Cms.Core.Sync.ServerRole`:

| Role | Responsibility |
|------|----------------|
| `SchedulingPublisher` | The "primary". Runs scheduled tasks. **Exactly one** app must hold this role. |
| `Subscriber` | Serves requests, does **not** run scheduled tasks, picks up cache instructions from the DB. |
| `Single` | One app does everything (default for standalone hosting). |

Cache stays in sync **automatically through the database** (`umbracoCacheInstruction` table) ÔÇö each app polls for cache instructions, so publishing on the backoffice refreshes NuCache on the public sites. No Redis / message bus is needed for a basic setup, only a shared database.

By default Umbraco auto-elects the publisher (`ElectedServerRoleAccessor`). To pin roles explicitly, set `uTPro:Hosting:ServerRole`. The template wires a config-driven `IServerRoleAccessor` in `Startup/UmbracoSetup.cs` (`ConfigureServerRole`):

Backoffice app:
```json
{
  "uTPro": {
    "Hosting": {
      "ServerRole": "SchedulingPublisher"
    }
  }
}
```

Public/website app(s):
```json
{
  "uTPro": {
    "Hosting": {
      "ServerRole": "Subscriber"
    }
  }
}
```

| Value | Effect |
|-------|--------|
| `SchedulingPublisher` / `Subscriber` / `Single` | Pins the role for this app, bypassing Umbraco's automatic election. |
| `""` (empty, default) or `Auto` | Leaves Umbraco's default database election in place. |

- **Exactly one** app in the cluster may be `SchedulingPublisher`; the rest must be `Subscriber`.
- An invalid value fails fast at startup with a clear exception (rather than silently mis-electing).
- Setting the role to `Subscriber` automatically makes that app **skip work that only the publisher should own** on a shared DB: it overrides `uSync:Settings:ImportOnFirstBoot` **and** `Umbraco:CMS:Unattended:UpgradeUnattended` to `false`. So only the `SchedulingPublisher` imports uSync / runs schema migrations, and subscribers can't race or duplicate them. Deploy the publisher first so the DB is upgraded before subscribers boot.

## 4.7.3 Sharing uploaded media between apps (`uTPro:Hosting:MediaPath`)

When you split a backoffice app and one or more website apps, uploaded files (logos, images, documents) must live in **one physical folder** so a file uploaded in the backoffice is served by every website. Everything else ÔÇö content, which media node is the logo, all "data" ÔÇö already lives in the **shared database**, so it does not need file-level sharing.

Point the physical media folder at a shared location with `uTPro:Hosting:MediaPath`, resolved in `Startup/HostingSetup.cs` (`ConfigureMediaPath`). It feeds Umbraco's `Umbraco:CMS:Global:UmbracoMediaPhysicalRootPath`:

```json
{
  "uTPro": {
    "Hosting": {
      "MediaPath": "D:\\Shared\\uTPro\\media"
    }
  }
}
```

- Empty string (default) Ôćĺ `~/media` under each app's `wwwroot`.
- Absolute paths are used as-is; relative paths resolve against `ContentRoot`.
- The folder is auto-created on boot; the app pool identity needs `Modify` on it.
- A non-empty value **overrides** any `Umbraco:CMS:Global:UmbracoMediaPhysicalRootPath` set elsewhere (it is applied last in the configuration chain). Leave `MediaPath` empty if you prefer to set the Umbraco native key directly.

> Prefer `MediaPath` over sharing the whole `wwwroot`: `App_Plugins` and compiled assets are per-app and should **not** be shared.

## 4.7.4 Sharing the whole `wwwroot` (`uTPro:Hosting:RootPath`)

`uTPro:Hosting:RootPath` relocates the entire web root (`wwwroot`). It is applied via `WebApplicationOptions` at startup (`Startup/HostingSetup.cs` Ôćĺ `BuildWebApplicationOptions`), because `WebApplicationBuilder` does not allow changing the web root after the builder is created. Use this **only when multiple apps deploy the exact same code** and you want them to share all static assets. For the common backoffice-plus-website split, prefer `MediaPath` (┬ž4.7.3) instead ÔÇö sharing only media avoids `App_Plugins` / bundle conflicts.

```json
{
  "uTPro": {
    "Hosting": {
      "RootPath": "D:\\Shared\\uTPro\\wwwroot"
    }
  }
}
```

- Empty string (default) Ôćĺ the app's own `./wwwroot`.
- Absolute paths are used as-is; relative paths resolve against `ContentRoot`.

## 4.7.5 Load-balancing checklist

| Concern | Setting |
|---------|---------|
| Database | Single shared DB for all apps (**required**) |
| Server role | Backoffice = `uTPro:Hosting:ServerRole = SchedulingPublisher`; websites = `Subscriber` (┬ž4.7.2) |
| Uploaded media | Shared folder via `uTPro:Hosting:MediaPath` (┬ž4.7.3) |
| TEMP / NuCache / Examine | **Separate per app** ÔÇö never shared (┬ž4.7, ┬ž4.7.1.1) |
| Data Protection keys | Shared path + shared application name via `uTPro:DataProtection:*` when apps/instances must validate each other's auth cookies / antiforgery tokens (┬ž4.7.6) |
| Splitting config per app | Shared base `appsettings.json` + per-app overlay `appsettings.backoffice.json` / `appsettings.website.json`, selected by `UTPRO_APP` (┬ž4.7.7) |

## 4.7.6 Data Protection keys (`uTPro:DataProtection:*`)

ASP.NET Core encrypts auth cookies and antiforgery tokens with **Data Protection keys**. By default (`KeysPath` empty + `LocalTempStorageLocation: EnvironmentTemp`) uTPro stores them under `%TEMP%/UmbracoData/<hash-of-ContentRoot>/PersistKeys`, and the hash is derived from **each app's own content root** ÔÇö so every app/instance gets a **different** key ring.

That is fine for a single instance, but it is the classic load-balancing pitfall the moment you run **more than one instance of the same app** behind a balancer: a request encrypted on instance A cannot be decrypted on instance B, so **antiforgery validation and auth cookies fail intermittently** (random 400s on form posts, users getting logged out). Resolved in `Startup/SecuritySetup.cs`.

To share keys, point every instance at the **same folder** and give them the **same application name**:

```json
{
  "uTPro": {
    "DataProtection": {
      "KeysPath": "D:\\Shared\\uTPro\\dp-keys",
      "ApplicationName": "uTPro"
    }
  }
}
```

| Key | Description |
|-----|-------------|
| `uTPro:DataProtection:KeysPath` | Shared folder for the key ring. Empty (default) Ôćĺ per-app path under `%TEMP%`. Absolute paths used as-is; relative paths resolve against `ContentRoot`. |
| `uTPro:DataProtection:ApplicationName` | Isolation boundary ÔÇö **must match** across all instances that share keys. Defaults to `uTPro`. |

- All instances sharing keys must use the **same** `KeysPath` **and** `ApplicationName`.
- Keys rotate every 90 days automatically; the shared folder needs `Modify` for every app pool identity.
- Whether to share keys **across** the backoffice and website apps depends on your topology: share them only if the two apps need to validate each other's cookies/tokens. Multiple instances of the *same* app always need to share.

## 4.7.7 Splitting config into shared + per-app overlays

When you run the same deployment as **two apps** (a backoffice app and a website app), most settings are identical ÔÇö database, media path, SMTP, features ÔÇö and only a few differ (server role, temp folder, backoffice domain toggle). Rather than maintaining two full copies of `appsettings.json`, keep **one shared base** and layer a small **per-app overlay** on top.

- `appsettings.json` ÔÇö the shared base (database, `uTPro:Hosting:MediaPath`, `uTPro:DataProtection:*`, Umbraco CMS settings, features). Used by **every** app. On its own it still runs a single standalone app unchanged.
- `appsettings.backoffice.json` ÔÇö backoffice-only overrides.
- `appsettings.website.json` ÔÇö website-only overrides.

The overlay is selected at startup in `Program.cs` by the **`UTPRO_APP`** environment variable (falling back to the `uTPro:App` config key):

```
UTPRO_APP=backoffice   Ôćĺ loads appsettings.json + appsettings.backoffice.json
UTPRO_APP=website      Ôćĺ loads appsettings.json + appsettings.website.json
(unset)                Ôćĺ loads appsettings.json only (standalone / single app)
```

An environment-specific overlay is also honoured if present: `appsettings.{app}.{Environment}.json` (e.g. `appsettings.website.Production.json`).

**What ships in each overlay** (template defaults ÔÇö adjust paths per deployment):

`appsettings.backoffice.json`:
```json
{
  "uTPro": {
    "Hosting": {
      "ServerRole": "SchedulingPublisher",
      "TempPath": "D:\\UmbracoTemp\\Backoffice"
    },
    "Backoffice": { "Enabled": true, "Url": "bo.utpro.local" }
  }
}
```

`appsettings.website.json`:
```json
{
  "uTPro": {
    "Hosting": {
      "ServerRole": "Subscriber",
      "TempPath": "D:\\UmbracoTemp\\Website"
    },
    "Backoffice": { "Enabled": false }
  }
}
```

**How to set `UTPRO_APP`:**

- **IIS**: `web.config` Ôćĺ `<aspNetCore><environmentVariables><environmentVariable name="UTPRO_APP" value="website" /></environmentVariables></aspNetCore>`, or set it on the app pool.
- **Local dev**: add it to the profile in `Properties/launchSettings.json` (`"environmentVariables": { "UTPRO_APP": "backoffice" }`).
- **Containers / CLI**: a normal environment variable.

**What goes where ÔÇö the rule:** a value that **differs between apps** belongs in the overlay; a value that is **identical for every app** stays in the shared `appsettings.json`. So `Hosting:ServerRole`, `Hosting:TempPath` and `Backoffice` (enabled on the backoffice app, disabled on the website app) live only in the overlays. Feature defaults such as `RememberLanguage` or `Feature:Form:RateLimit` are the same everywhere, so they stay shared even though only one app reads them.

- `uTPro:Backoffice` is **not** in the shared base. When no overlay is applied (single-app / standalone), the key is absent, so backoffice-domain separation defaults to **off** ÔÇö the backoffice is served on the main domain, the standard Umbraco behaviour. Local dev keeps its own values in `appsettings.Development.json`.

**Precedence:** the overlay is added **after** the default configuration sources, so it overrides `appsettings.json`, `appsettings.{Environment}.json` **and** environment variables. `TempPath` must be **different** for each app (never share the temp folder ÔÇö see ┬ž4.7).

## 4.7.8 Deployment topology ÔÇö one folder per app

The per-app overlay (┬ž4.7.7) and the automatic per-app temp / Data Protection isolation both key off the app's **content root path**. So each app **must be its own deployment folder** with its own content root. Do **not** point two IIS sites at a single physical folder ÔÇö that breaks all three mechanisms:

1. **Overlay selection** ÔÇö two sites sharing one folder share one `web.config`, so you can't give them different `UTPRO_APP` values there.
2. **Temp collision** ÔÇö with `TempPath` empty + `LocalTempStorageLocation: EnvironmentTemp`, both Umbraco and `SecuritySetup` derive the temp folder from a hash of the content root. Same folder Ôćĺ same hash Ôćĺ same temp Ôćĺ corrupt NuCache / Lucene (see ┬ž4.7).
3. **Data Protection key collision** ÔÇö the fallback key path is hashed from the content root the same way.

**Correct layout** ÔÇö two independent deployments of the same build, sharing state through the database, the media folder and the DP key folder:

```
D:\sites\backoffice\    ÔćÉ content root A, web.config sets UTPRO_APP=backoffice
D:\sites\website\       ÔćÉ content root B, web.config sets UTPRO_APP=website

D:\shared\media\        ÔćÉ uTPro:Hosting:MediaPath   (shared, both apps)
D:\shared\dp-keys\      ÔćÉ uTPro:DataProtection:KeysPath (shared, both apps)
D:\UmbracoTemp\Backoffice\   ÔćÉ uTPro:Hosting:TempPath (per app, NOT shared)
D:\UmbracoTemp\Website\      ÔćÉ uTPro:Hosting:TempPath (per app, NOT shared)
```

| Resource | Shared or per-app |
|----------|-------------------|
| Deployment folder / content root | **Per app** |
| `web.config` (`UTPRO_APP`) | **Per app** |
| `TempPath`, Examine index, NuCache | **Per app** |
| Database | **Shared** |
| `MediaPath` (uploaded files) | **Shared** |
| `DataProtection:KeysPath` (+ `ApplicationName`) | **Shared** (when apps must validate each other's tokens) |

**Set `UTPRO_APP` in each site's `web.config`:**

```xml
<aspNetCore processPath="dotnet" arguments=".\uTPro.Project.Web.dll" hostingModel="inprocess">
  <environmentVariables>
    <environmentVariable name="UTPRO_APP" value="website" />
  </environmentVariables>
</aspNetCore>
```

**Boot order:** start the `SchedulingPublisher` (backoffice) app first so it runs schema migrations and the uSync first-boot import against the shared DB; then start the `Subscriber` (website) apps. Subscribers skip migrations/import by design (┬ž4.7.2), so a subscriber booting against an un-migrated DB would fail or serve an empty site.

# 4.8 Database Provider (PostgreSQL / SQL Server / SQLite)

uTPro runs on any database engine Umbraco 17 supports. The engine is selected by two keys under `ConnectionStrings` in `appsettings.json`: `umbracoDbDSN` (the connection string) and `umbracoDbDSN_ProviderName` (the ADO.NET provider).

uTPro ships configured for **PostgreSQL** out of the box ÔÇö `uTPro.Project.Web` references the `Our.Umbraco.PostgreSql` package and both `appsettings.json` / `appsettings.Development.json` default to the `Npgsql2` provider. Switching to SQL Server or SQLite requires only changing the two connection-string keys below; no code changes.

The PostgreSQL provider is registered **conditionally** in `Startup/UmbracoSetup.cs`: `AddUmbracoPostgreSqlSupport()` is only called when `umbracoDbDSN_ProviderName` contains `Npgsql` (case-insensitive, so it matches both `Npgsql` and `Npgsql2`). When you switch to SQL Server or SQLite, that call is skipped automatically ÔÇö the PostgreSQL provider never gets wired up.

| Engine | `umbracoDbDSN_ProviderName` | Extra package required |
|--------|-----------------------------|------------------------|
| PostgreSQL (default) | `Npgsql2` | `Our.Umbraco.PostgreSql` (pre-referenced) |
| SQL Server | `Microsoft.Data.SqlClient` | none (built into Umbraco) |
| SQLite | `Microsoft.Data.Sqlite` | none (built into Umbraco) |

### PostgreSQL (default)

Create an empty PostgreSQL database, then point the connection string at it:

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=localhost;Port=5432;Database=uTPro;User Id=postgres;Password=postgres;",
    "umbracoDbDSN_ProviderName": "Npgsql2"
  }
}
```

### SQL Server

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=(local);Database=uTPro;Integrated Security=true;TrustServerCertificate=true;",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.SqlClient"
  }
}
```

### SQLite

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Data Source=|DataDirectory|/Umbraco.sqlite.db;Cache=Shared;Foreign Keys=True;Pooling=True",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.Sqlite"
  }
}
```

> **Note:** The `Our.Umbraco.PostgreSql` package is a community provider ([Our.Umbraco.PostgreSql on NuGet](https://www.nuget.org/packages/Our.Umbraco.PostgreSql)). Its provider name is `Npgsql2` (not `Npgsql`) ÔÇö this is intentional, so it doesn't clash with the stock `Npgsql` ADO.NET factory. Keep its version aligned with your Umbraco major version (e.g. `17.x` for Umbraco 17). Never mix providers in the same environment ÔÇö pick one engine per deployment. The default credentials (`postgres`/`postgres`) are for local development only; use a dedicated least-privilege role and a secret store for production. Leaving an empty `umbracoDbDSN` triggers the Umbraco install wizard, which lets you choose the engine from the UI instead.

# 4.9 Reverse proxy / Forwarded Headers (real client IP)

When uTPro runs **behind a reverse proxy or load balancer** (the typical Linux setup of Kestrel behind nginx/Apache, as well as cloud load balancers and Cloudflare), `HttpContext.Connection.RemoteIpAddress` is the *proxy's* IP, not the visitor's. That silently breaks every per-IP feature ÔÇö for example the [SimpleFormBuilder](https://github.com/T4VN/uTPro.Feature.SimpleFormBuilder) submission rate limiter would treat all visitors as one shared client.

uTPro configures ASP.NET Core's forwarded-headers processing in `Startup/ForwardedHeadersSetup.cs` so the app resolves the real client IP (and scheme) from the `X-Forwarded-For` / `X-Forwarded-Proto` headers. The middleware runs at the very front of the pipeline, so HTTPS redirection, rate limiting and logging all see the real client.

This concern lives in the **host** (uTPro), not in individual feature packages ÔÇö the host owns the deployment topology and therefore decides which proxies to trust.

**Disabled by default.** Direct-exposed Kestrel and IIS in-process deployments already surface the real client IP, so the feature is a no-op until you opt in. Enable it only when you actually sit behind a proxy.

```json
{
  "uTPro": {
    "ForwardedHeaders": {
      "Enabled": true,
      "KnownProxies": [ "10.0.0.5" ],
      "KnownNetworks": [ "10.0.0.0/8" ],
      "ForwardLimit": 1,
      "TrustAllProxies": false
    }
  }
}
```

| Key | Description |
|-----|-------------|
| `uTPro:ForwardedHeaders:Enabled` | Master switch. When `false` (default) the middleware is not registered. |
| `uTPro:ForwardedHeaders:KnownProxies` | Array of trusted proxy IP addresses (e.g. `"10.0.0.5"`). The defaults are cleared, so only the proxies you list here are trusted. |
| `uTPro:ForwardedHeaders:KnownNetworks` | Array of trusted networks in CIDR notation (e.g. `"10.0.0.0/8"`, `"2001:db8::/32"`). |
| `uTPro:ForwardedHeaders:ForwardLimit` | Number of proxy hops to walk back through `X-Forwarded-For`. Defaults to `1` (single reverse proxy); raise it for chained proxies such as CDN Ôćĺ nginx. |
| `uTPro:ForwardedHeaders:TrustAllProxies` | Trusts **every** proxy IP (`0.0.0.0/0` + `::/0`). For container / dynamic-proxy scenarios where the proxy IP isn't stable. See the warning below. |

> ÔÜá´ŞĆ **`TrustAllProxies` warning:** only enable it when the app is **not** directly reachable by clients (i.e. it can *only* be reached through your proxy). If the app is directly exposed, a client can spoof `X-Forwarded-For` to impersonate any IP and bypass per-IP protections such as rate limiting. Prefer listing `KnownProxies` / `KnownNetworks` whenever the proxy addresses are stable.

> **Tip:** if you deploy the SimpleFormBuilder (or any per-IP feature) behind a proxy, enabling this section is what makes per-IP rate limiting work correctly.

# 4.10 Responsive images

uTPro ships an `Html.ResponsiveImg(...)` Razor helper (`ResponsiveImageExtension`) that renders a modern responsive `<img>` ÔÇö automatic **WebP** conversion, a `srcset` bounded by each image's longest edge, `sizes`, intrinsic width/height to avoid layout shift (CLS), and `loading="lazy"` / `decoding="async"` by default.

Behaviour is driven per site by a **`GlobalResponsiveImageSettings`** node composed onto the **Folder Settings** node:

| Setting | Effect |
|---------|--------|
| `RepImgTurnOffResponsiveImage` | When **on**, responsive processing is enabled: each image is resized and converted to WebP to build the `srcset`. When **off** (or no settings node exists, or the media isn't an image), the original `MediaUrl()` is emitted untouched. |
| `Extend Sizes SrcSet` | The candidate widths for the `srcset`. When empty, the defaults **1200 / 1024 / 768** are used. Images are never upscaled beyond their intrinsic size. |

Helper parameters let a template opt into per-image behaviour:

```razor
@* Above-the-fold hero: eager load + fetchpriority="high" *@
@Model.HeroImage.ResponsiveImg(alt: "Hero", priority: true)

@* Standard lazy-loaded image with a custom sizes attribute *@
@Model.CardImage.ResponsiveImg(cssClass: "card-img", sizes: "(max-width: 768px) 100vw, 33vw")
```

- `alt` ÔÇö falls back to the media's `altText` property, then to an empty (decorative) alt.
- `priority` ÔÇö eager load + `fetchpriority="high"` for LCP images (overrides `lazy`).
- `includeDimensions` ÔÇö emit intrinsic width/height + responsive inline style to reserve space (default `true`).
- `quality` ÔÇö WebP quality (default `80`).
