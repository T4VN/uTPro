# 4.4 Security Configuration

uTPro includes built-in security hardening. Most of it is code-level (in `Startup/*.cs`), while the HTTP **security headers are content-driven** so editors can tune them per site from the backoffice without a redeploy:

- **Security Headers** — resolved per request by `SecurityHeadersFoundation` (in `uTPro.Foundation.Middleware`) from a per-site **`GlobalSecurityHeadersSettings`** backoffice node (see [4.4.1](#441-security-headers-backoffice-driven)). When that node is absent or its `Enabled` toggle is off, no security headers are emitted.
- **Request Size Limits**: 128MB max upload, 4MB per form value, 10240 max form values
- **Sync IO disabled**: Both IIS and Kestrel have `AllowSynchronousIO = false` to prevent thread-pool starvation under load
- **Data Protection**: Keys persisted with 90-day rotation, configurable shared path for load-balanced deployments
- **Domain Policy**: Configurable domain allowlist with wildcard support in `CheckPolicy.cs`
- **Kestrel**: Server header disabled

## 4.4.1 Security headers (backoffice-driven)

Security headers are no longer hard-coded — they are read from a `GlobalSecurityHeadersSettings` node that can be composed onto the site's **Folder Settings** node, the **site root**, or the **current page** (first match wins). Sensible code defaults fill in any field left blank on an enabled node, so turning the feature on with everything blank still gives you `nosniff`, `SAMEORIGIN`, and a 1-year HSTS on HTTPS.

| Field | Emitted header | Default when blank |
|-------|----------------|--------------------|
| `Enabled` | *(master toggle — off means no headers at all)* | — |
| `XContentTypeOptions` | `X-Content-Type-Options` | `nosniff` |
| `ReferrerPolicy` | `Referrer-Policy` | *(omitted)* |
| `PermissionsPolicy` | `Permissions-Policy` | *(omitted)* |
| `XFrameOptions` | `X-Frame-Options` | `SAMEORIGIN` (only when no CSP `frame-ancestors` is in play) |
| `ContentSecurityPolicyCsp` | `Content-Security-Policy` | *(omitted)* |
| `ContentSecurityPolicyReportOnly` | sends the CSP as `Content-Security-Policy-Report-Only` instead (framing stays enforced) | `false` |
| `HstsEnabled` | `Strict-Transport-Security` (HTTPS requests only) | `false` |
| `HstsMaxAgeSeconds` | `max-age=…` | `31536000` (1 year) |
| `HstsIncludeSubDomains` | `; includeSubDomains` | `false` |
| `HstsPreload` | `; preload` | `false` |

**Behaviour worth knowing:**

- **Clickjacking protection is always enforced** on an enabled node. When a backoffice domain is configured (`uTPro:Backoffice:Domain`), the trusted backoffice host is added to a CSP `frame-ancestors 'self' …` directive so cross-domain backoffice **preview keeps working**; otherwise `X-Frame-Options` is used. An explicit CSP that omits `frame-ancestors` gets the backoffice host appended automatically.
- **Header-injection safe**: every value is collapsed to a single line (CR/LF/tabs/multiple spaces → one space) before being emitted, so editors can format a CSP one directive per line without risking response splitting (CWE-113).
- **Fails open**: if the settings/content lookup errors, the request is served with no security headers (and a warning is logged) rather than returning a 500 — a content issue can never take the frontend down.

# 4.5 Performance Configuration

Performance is configured in `Startup/PerformanceSetup.cs` and `Startup/PipelineSetup.cs`:

### Minification & Compression
- **WebOptimizer**: CSS/JS minification for `css/**/*.css`, `assets/**/*.css`, `js/**/*.js`, `assets/**/*.js`, `scripts/**/*.js`
- **WebOptimizer forced in Development**: `webOptimizer.enableCaching: true` in `appsettings.Development.json` ensures minification runs in all environments
- **WebMarkupMin**: HTML minification + Brotli/GZip/Deflate compression (Brotli prioritized, 10MB max response size)
- **Statistics disabled** in production to save CPU
- **Pipeline order**: `UseWebOptimizer()` → `UseStaticFiles()` → `UseWebMarkupMin()` (critical for minification to work)

### Caching (multi-layer)
- **Output Cache** (`AddOutputCache`): Stores fully-rendered HTML responses in server memory. Subsequent requests skip Razor + Umbraco entirely (0ms server processing). Policies:
  - `Page` — 120s TTL, varies by host/query/Accept-Language, tagged `"pages"` for eviction
  - `Sitemap` — 30min TTL, varies by host, tagged `"sitemap"`
- **Browser Cache** (`Cache-Control` header): Website pages get `public, max-age=120, stale-while-revalidate=60` — browser shows cached page instantly while revalidating in background
- **Static File Cache**:
  - Assets in `/assets/`, `/css/`, `/scripts/`, `/js/` → `public, max-age=31536000, immutable` (1 year)
  - Media uploads in `/uploads/`, `/media/` → `public, max-age=604800, stale-while-revalidate=86400` (7 days)
- **Sitemap Memory Cache** (`IMemoryCache`): XML sitemap cached 30 minutes in `SitemapFoundation`
- **Response Caching**: Adds `Vary` headers for downstream CDN/proxy validation

### Razor
- **Runtime Compilation**: Only enabled in Development environment (saves ~200MB RAM in production)
- **Precompiled views**: `RazorCompileOnBuild=true` in csproj ensures views are compiled at build time

### Request Buffering
- **Global buffering disabled**: `BufferBody = false` — no longer copies every request body into memory
- **Selective buffering**: Only endpoints that need to re-read the body should enable it explicitly

# 4.7 Multi-site IIS hosting (load balancing) — moving `umbraco/Data/TEMP` out of the project

When you host one backoffice + multiple public sites on IIS that share the same SQL database, do **not** try to share `umbraco/Data/TEMP` between them. NuCache, Examine indexes and `MainDom` all take exclusive file locks, so a shared folder causes boot failures and corrupt indexes.

The correct pattern is the same one Umbraco documents for [Azure Web Apps load balancing](https://docs.umbraco.com/umbraco-cms/fundamentals/setup/server-setup/load-balancing/azure-web-apps):

- Each site keeps its own TEMP folder, placed in `%TEMP%` (outside the project).
- Content changes propagate via `DatabaseServerMessenger` (polls the shared SQL).
- The backoffice site is pinned as `SchedulingPublisher`; public sites are `Subscriber`s.

## 4.7.1 Baseline settings (apply to every site)

Add these three settings to every site's `appsettings.json`. The template already ships them configured:

```json
{
  "Umbraco": {
    "CMS": {
      "Global": {
        "MainDomLock": "FileSystemMainDomLock"
      },
      "Hosting": {
        "LocalTempStorageLocation": "EnvironmentTemp"
      },
      "Examine": {
        "LuceneDirectoryFactory": "TempFileSystemDirectoryFactory"
      }
    }
  }
}
```

| Key | Effect |
|---|---|
| `Hosting:LocalTempStorageLocation: EnvironmentTemp` | Umbraco writes NuCache, locks and other TEMP files to `%TEMP%/UmbracoData/<hash-of-ContentRoot>/`. Each site gets its own folder automatically, nothing lands in the project. |
| `Examine:LuceneDirectoryFactory: TempFileSystemDirectoryFactory` | Lucene indexes are created under the same `%TEMP%` location. Use `SyncedTempFileSystemDirectoryFactory` instead on the single-instance backoffice if you want indexes persisted and synced on restart. |
| `Global:MainDomLock: FileSystemMainDomLock` | File-based MainDom lock. Works across multiple processes / instances on Windows where the default semaphore is machine-wide and breaks multi-site hosting. |

On IIS make sure the app pool identity has write access to `%TEMP%`. With `ApplicationPoolIdentity` that is `C:\Windows\Temp` by default (enable "Load User Profile" only if you actually need a per-profile temp).

> Media uploads (`uploads/`) are controlled separately via `Umbraco:CMS:Global:UmbracoMediaPath` and are not affected by these settings.

### 4.7.1.1 Override `%TEMP%` from `appsettings.json` (code-driven alternative)

If you don't want to touch `web.config` on every deploy, this template provides a source-code override in `Startup/HostingSetup.cs`. Set `uTPro:Hosting:TempPath` and the app will set the process-level `TMP` / `TEMP` environment variables at startup, before Umbraco reads `Path.GetTempPath()`:

```json
{
  "uTPro": {
    "Hosting": {
      "TempPath": "D:\\UmbracoTemp\\SiteA"
    }
  }
}
```

- Empty string (the default) leaves `TMP` / `TEMP` untouched — Umbraco falls back to the OS-provided temp folder.
- Absolute paths are used as-is.
- Relative paths are resolved against `ContentRoot` (NOT the process working directory).
- The folder is auto-created on boot if missing; the app pool identity must have `Modify` on its parent.

# 4.8 Database Provider (PostgreSQL / SQL Server / SQLite)

uTPro runs on any database engine Umbraco 17 supports. The engine is selected by two keys under `ConnectionStrings` in `appsettings.json`: `umbracoDbDSN` (the connection string) and `umbracoDbDSN_ProviderName` (the ADO.NET provider).

uTPro ships configured for **PostgreSQL** out of the box — `uTPro.Project.Web` references the `Our.Umbraco.PostgreSql` package and both `appsettings.json` / `appsettings.Development.json` default to the `Npgsql2` provider. Switching to SQL Server or SQLite requires only changing the two connection-string keys below; no code changes.

The PostgreSQL provider is registered **conditionally** in `Startup/UmbracoSetup.cs`: `AddUmbracoPostgreSqlSupport()` is only called when `umbracoDbDSN_ProviderName` contains `Npgsql` (case-insensitive, so it matches both `Npgsql` and `Npgsql2`). When you switch to SQL Server or SQLite, that call is skipped automatically — the PostgreSQL provider never gets wired up.

| Engine | `umbracoDbDSN_ProviderName` | Extra package required |
|--------|-----------------------------|------------------------|
| PostgreSQL (default) | `Npgsql2` | `Our.Umbraco.PostgreSql` (pre-referenced) |
| SQL Server | `Microsoft.Data.SqlClient` | none (built into Umbraco) |
| SQLite | `Microsoft.Data.Sqlite` | none (built into Umbraco) |

### PostgreSQL (default)

Create an empty PostgreSQL database, then point the connection string at it:

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=localhost;Port=5432;Database=uTPro;User Id=postgres;Password=postgres;",
    "umbracoDbDSN_ProviderName": "Npgsql2"
  }
}
```

### SQL Server

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Server=(local);Database=uTPro;Integrated Security=true;TrustServerCertificate=true;",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.SqlClient"
  }
}
```

### SQLite

```json
{
  "ConnectionStrings": {
    "umbracoDbDSN": "Data Source=|DataDirectory|/Umbraco.sqlite.db;Cache=Shared;Foreign Keys=True;Pooling=True",
    "umbracoDbDSN_ProviderName": "Microsoft.Data.Sqlite"
  }
}
```

> **Note:** The `Our.Umbraco.PostgreSql` package is a community provider ([Our.Umbraco.PostgreSql on NuGet](https://www.nuget.org/packages/Our.Umbraco.PostgreSql)). Its provider name is `Npgsql2` (not `Npgsql`) — this is intentional, so it doesn't clash with the stock `Npgsql` ADO.NET factory. Keep its version aligned with your Umbraco major version (e.g. `17.x` for Umbraco 17). Never mix providers in the same environment — pick one engine per deployment. The default credentials (`postgres`/`postgres`) are for local development only; use a dedicated least-privilege role and a secret store for production. Leaving an empty `umbracoDbDSN` triggers the Umbraco install wizard, which lets you choose the engine from the UI instead.

# 4.9 Reverse proxy / Forwarded Headers (real client IP)

When uTPro runs **behind a reverse proxy or load balancer** (the typical Linux setup of Kestrel behind nginx/Apache, as well as cloud load balancers and Cloudflare), `HttpContext.Connection.RemoteIpAddress` is the *proxy's* IP, not the visitor's. That silently breaks every per-IP feature — for example the [SimpleFormBuilder](https://github.com/T4VN/uTPro.Feature.SimpleFormBuilder) submission rate limiter would treat all visitors as one shared client.

uTPro configures ASP.NET Core's forwarded-headers processing in `Startup/ForwardedHeadersSetup.cs` so the app resolves the real client IP (and scheme) from the `X-Forwarded-For` / `X-Forwarded-Proto` headers. The middleware runs at the very front of the pipeline, so HTTPS redirection, rate limiting and logging all see the real client.

This concern lives in the **host** (uTPro), not in individual feature packages — the host owns the deployment topology and therefore decides which proxies to trust.

**Disabled by default.** Direct-exposed Kestrel and IIS in-process deployments already surface the real client IP, so the feature is a no-op until you opt in. Enable it only when you actually sit behind a proxy.

```json
{
  "uTPro": {
    "ForwardedHeaders": {
      "Enabled": true,
      "KnownProxies": [ "10.0.0.5" ],
      "KnownNetworks": [ "10.0.0.0/8" ],
      "ForwardLimit": 1,
      "TrustAllProxies": false
    }
  }
}
```

| Key | Description |
|-----|-------------|
| `uTPro:ForwardedHeaders:Enabled` | Master switch. When `false` (default) the middleware is not registered. |
| `uTPro:ForwardedHeaders:KnownProxies` | Array of trusted proxy IP addresses (e.g. `"10.0.0.5"`). The defaults are cleared, so only the proxies you list here are trusted. |
| `uTPro:ForwardedHeaders:KnownNetworks` | Array of trusted networks in CIDR notation (e.g. `"10.0.0.0/8"`, `"2001:db8::/32"`). |
| `uTPro:ForwardedHeaders:ForwardLimit` | Number of proxy hops to walk back through `X-Forwarded-For`. Defaults to `1` (single reverse proxy); raise it for chained proxies such as CDN → nginx. |
| `uTPro:ForwardedHeaders:TrustAllProxies` | Trusts **every** proxy IP (`0.0.0.0/0` + `::/0`). For container / dynamic-proxy scenarios where the proxy IP isn't stable. See the warning below. |

> ⚠️ **`TrustAllProxies` warning:** only enable it when the app is **not** directly reachable by clients (i.e. it can *only* be reached through your proxy). If the app is directly exposed, a client can spoof `X-Forwarded-For` to impersonate any IP and bypass per-IP protections such as rate limiting. Prefer listing `KnownProxies` / `KnownNetworks` whenever the proxy addresses are stable.

> **Tip:** if you deploy the SimpleFormBuilder (or any per-IP feature) behind a proxy, enabling this section is what makes per-IP rate limiting work correctly.

# 4.10 Responsive images

uTPro ships an `Html.ResponsiveImg(...)` Razor helper (`ResponsiveImageExtension`) that renders a modern responsive `<img>` — automatic **WebP** conversion, a `srcset` bounded by each image's longest edge, `sizes`, intrinsic width/height to avoid layout shift (CLS), and `loading="lazy"` / `decoding="async"` by default.

Behaviour is driven per site by a **`GlobalResponsiveImageSettings`** node composed onto the **Folder Settings** node:

| Setting | Effect |
|---------|--------|
| `RepImgTurnOffResponsiveImage` | When **on**, responsive processing is enabled: each image is resized and converted to WebP to build the `srcset`. When **off** (or no settings node exists, or the media isn't an image), the original `MediaUrl()` is emitted untouched. |
| `Extend Sizes SrcSet` | The candidate widths for the `srcset`. When empty, the defaults **1200 / 1024 / 768** are used. Images are never upscaled beyond their intrinsic size. |

Helper parameters let a template opt into per-image behaviour:

```razor
@* Above-the-fold hero: eager load + fetchpriority="high" *@
@Model.HeroImage.ResponsiveImg(alt: "Hero", priority: true)

@* Standard lazy-loaded image with a custom sizes attribute *@
@Model.CardImage.ResponsiveImg(cssClass: "card-img", sizes: "(max-width: 768px) 100vw, 33vw")
```

- `alt` — falls back to the media's `altText` property, then to an empty (decorative) alt.
- `priority` — eager load + `fetchpriority="high"` for LCP images (overrides `lazy`).
- `includeDimensions` — emit intrinsic width/height + responsive inline style to reserve space (default `true`).
- `quality` — WebP quality (default `80`).
