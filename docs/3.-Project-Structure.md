# 3. Project Structure

## 3.1 Prerequisites

- **.NET 10 SDK** (10.0.301 or later — pinned in `global.json`)
- **A database engine** — **PostgreSQL** (default), **SQL Server**, or **SQLite** (see [4.8](4.-Configurations#48-database-provider-postgresql--sql-server--sqlite))
- **Visual Studio 2022** (v17.12+) or **VS Code** with C# Dev Kit
- **Node.js** (optional, for frontend tooling)

## 3.2 Solution Architecture

uTPro follows a clean modular architecture with clear separation of concerns:

```
uTPro/
├── Common/
│   └── uTPro.Common/              # Shared models, constants, CMS content types
├── Extension/
│   ├── uTPro.Extension/           # Reusable helpers (HTML sections, script queue, content traversal, media URLs)
│   └── uTPro.Extension.CurrentSite/ # Per-request site context (culture, domains, navigation)
├── Foundation/
│   ├── uTPro.Foundation/          # Media path scheme
│   ├── uTPro.Foundation.Favicon/  # Dynamic favicon from CMS
│   ├── uTPro.Foundation.Middleware/ # Request localization, buffering
│   ├── uTPro.Foundation.Robots/   # robots.txt generation
│   └── uTPro.Foundation.Sitemap/  # sitemap.xml generation (cached)
├── Feature/
│   ├── uTPro.Feature/             # Feature base (shared partials, e.g. SimpleForm views)
│   └── uTPro.Feature.Dashboard/   # Backoffice dashboard + header app (see docs 6)
├── Project/
│   ├── uTPro.Project.Web/         # Main web application
│   └── uTPro.Project.Web.Configure/ # Render controller, error handling, policies
├── global.json
└── uTPro.sln
```

> **Optional feature packages** ship as standalone NuGet packages in their own repositories (not part of this solution): **File Manager**, **Simple Form Builder**, **Audit Log**, **Job Monitor**, **Auto Translation**, and **URL Viewer**. Add only the ones you need.

## 3.4 Program.cs — Clean Entry Point

The application entry point is intentionally minimal. All configuration logic is organized into extension methods in the `Startup/` folder:

```csharp
using uTPro.Common.Constants;
using uTPro.Project.Web.Configure;
using uTPro.Project.Web.Startup;

// Web root (uTPro:Hosting:RootPath) must be applied via options.
var setup = HostingSetup.BuildWebApplicationOptions(args);
var builder = WebApplication.CreateBuilder(setup);

// Expose the resolved web/content root to the DI-less PathFolder helper.
PathFolder.WebRootPathOverride = builder.Environment.WebRootPath;
PathFolder.ContentRootPathOverride = builder.Environment.ContentRootPath;

builder.ConfigureAppSettings();                               // Per-app overlay (UTPRO_APP)
builder.ConfigureTempPath();                                  // Hosting: TMP/TEMP
builder.ConfigureMediaPath();                                 // Hosting: shared media
builder.ConfigureUmbraco();                                   // Umbraco CMS
builder.Services.AddPerformanceServices(builder.Environment); // Performance
builder.Services.AddSecurityServices(builder);                // Security
builder.Services.AddForwardedHeadersConfig(builder);          // Real client IP/scheme
builder.Services.AddRenderingDefaults();                      // MVC
builder.Services.AddControllers();
builder.Services.AddBlockPreviewServices();                   // Block Preview

var app = builder.Build();
await app.BootUmbracoAsync();
app.ConfigurePipeline();                                      // Middleware
await app.RunAsync();
```

Each `Startup/*.cs` file is self-contained and can be modified independently.

## 3.5 Middleware Pipeline Order

Configured in `Startup/PipelineSetup.cs`:

1. **Forwarded Headers** — rewrites client IP/scheme from `X-Forwarded-*` (must run first so HTTPS redirect, rate limiting and logging see the real client)
2. **Browser Cache-Control** — `stale-while-revalidate` for HTML pages
3. **HTTPS Redirection**
4. **Status Code Pages** — `UseStatusCodePagesWithReExecute("/error/{0}")` (preserves 404/500 status)
5. **WebOptimizer** — CSS/JS minification (must be before StaticFiles)
6. **Static Files** — immutable 1-year cache for `/assets/`, `/css/`, `/scripts/`, `/js/`; 7-day cache for `/uploads/`
7. **WebMarkupMin** — HTML minification + Brotli/GZip compression
8. **Output Cache** — stores rendered responses in memory (120s TTL)
9. **Cookie Policy**
10. **uTPro Middleware** — request localization (culture cookie) + selective buffering
11. **Umbraco Pipeline** — BackOffice + Website routing. **Security headers** run here (PostRouting), self-registered via `SecurityHeadersComposer` and driven per-site by the `GlobalSecurityHeadersSettings` node (see [4.4.1](4.-Configurations#441-security-headers-backoffice-driven))
12. **MapControllers** — MVC endpoint routing

> **Important:** `UseWebOptimizer()` must come before `UseStaticFiles()`. WebOptimizer intercepts CSS/JS requests to minify them. If StaticFiles runs first, it serves the original unminified file and WebOptimizer never sees the request.

## 3.7 Setup

### Database
uTPro ships configured for **PostgreSQL** (`umbracoDbDSN_ProviderName: "Npgsql2"`). Create an empty database on your chosen engine (PostgreSQL, SQL Server, or SQLite), then update `umbracoDbDSN` and `umbracoDbDSN_ProviderName` in `appsettings.json`. See [4.8](4.-Configurations#48-database-provider-postgresql--sql-server--sqlite) for each engine's connection string and provider name.

### Build & Run
```bash
dotnet build uTPro/uTPro.sln
dotnet run --project uTPro/Project/uTPro.Project.Web/uTPro.Project.Web.csproj
```

Navigate to `https://localhost:44381/umbraco` to complete Umbraco installation.

### Setup Data
The project includes uSync data. After installing Umbraco CMS, go to the uSync dashboard → "Everything" group → "Import" to synchronize all content, document types, and settings.

## 3.8 Setup Domain

For multi-site or separate backoffice configuration, update the **hosts file**:

- Windows: `C:\Windows\System32\Drivers\etc\hosts`
- MacOS/Linux: `/etc/hosts`

Example:
```
127.0.0.1   utpro.local
127.0.0.1   bo.utpro.local
```

## 3.9 Script Queue — Loading JS from Block Components

Block grid components often need their own JavaScript. uTPro provides a **Script Queue** system (`ScriptQueueExtensions`) that lets components register JS files, and the Layout renders them in the correct loading position.

### Why not just use `<script>` in the component?

The Layout uses a **deferred loader** that loads jQuery + plugins only after user interaction (or 3s idle). If a component renders its own `<script defer>` tag, it may execute **before** jQuery is available — causing `$ is not defined` errors.

### Two queues

| Method | When to use | How it loads |
|--------|-------------|--------------|
| `Html.QueueScript(path)` | Script **needs jQuery** or other deferred libraries | Appended to the sequential deferred loader array (loads after jQuery) |
| `Html.QueueStandaloneScript(path)` | Script has **no dependencies** (vanilla JS) | Rendered as a separate `<script defer>` tag |

Both methods **deduplicate automatically** — if the same component appears twice on a page, the script is only loaded once.

### Usage in a component `.cshtml`

```razor
@* Script that needs jQuery *@
@{ Html.QueueScript($"/scripts/{CurrentSite.GetItem().Root.Name}/slider.js"); }

@* Standalone script (no jQuery needed) *@
@{ Html.QueueStandaloneScript($"/scripts/{CurrentSite.GetItem().Root.Name}/cookie-consent-banner.js"); }
```

### What happens in the Layout

```html
<!-- Deferred loader (jQuery + plugins + component scripts via QueueScript) -->
<script>
    var scripts = [
        '/assets/scripts/jquery.min.js',
        ...
        '/scripts/uTPro/main.js',
        '/scripts/uTPro/slider.js'   ← added by component via QueueScript
    ];
    // loads sequentially after interaction/idle
</script>

<!-- Standalone scripts (via QueueStandaloneScript) -->
<script src="/scripts/uTPro/cookie-consent-banner.js" defer></script>
```

### Rules

1. **Vanilla JS** (no dependencies) → use `QueueStandaloneScript`
2. **Needs jQuery** → use `QueueScript`
3. **Never** put `<script src="...">` directly in a component — use the queue instead
4. CSS can still use `SetSection` as before (CSS doesn't have dependency ordering issues)
